## 1.2. XSS przez `eval` w `CookieService.SetAsync`

`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions.Website/Cookies/Services/CookieService.cs:68`

```csharp
await _runtime.InvokeVoidAsync("eval", $"document.cookie = '{cookieString}';");
```

**JavaScript injection sink**. Jeśli `cookie.Name` lub `cookie.Value` zawiera `'`, backtick, `</script>`, lub specjalne znaki → arbitrary code execution. Przykład:

```

```

**Naprawa**: zarejestrować JS helper i przekazywać wartości jako parametry:

```csharp
await _runtime.InvokeVoidAsync("zonit.setCookie", cookie.Name, cookie.Value, opts);
```

## 1.4. URL-prefix culture detection po `UseRouting` — routing nie widzi przepisanej ścieżki

`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions.Website/DependencyInjection/WebsiteServiceCollectionExtensions.cs:309-322`

```
branch.UseRouting();          // ← endpoint już zmatchowany dla /pl/home
branch.UseAuthentication();
branch.UseAuthorization();
...
branch.UseMiddleware<CultureMiddleware>();   // ← przepisuje Path = "/home", ale za późno
```

`CultureMiddleware.ResolveCulture` przepisuje `httpContext.Request.Path` z `/pl/home` na `/home`. Ale `UseRouting` (a właściwie `EndpointRoutingMiddleware` które ono rejestruje) już wybrał endpoint (najpewniej żaden — 404 fallback) bo `/pl/home` nie pasuje do żadnej trasy. Nawet jeśli `UseEndpoints` widzi przepisaną ścieżkę, **endpoint jest już ustalony** przez `EndpointFeature`.

**Naprawa** — wymaga decyzji designerskiej, nie samego przesunięcia:

1. **Opcja A: `UseRewriter`** z `IRewriteRule` przed `UseRouting`. Czysty pattern, ale `CultureMiddleware` traci dostęp do `RouteValues`/`EndpointFeature` (których obecnie nie używa, ale to ogranicza future expansion).
2. **Opcja B: middleware przed `UseRouting`**, ale wtedy `CultureMiddleware` musi tylko przepisać `Path` i ustawić cookie, a wszystko inne (binding na scoped `ICultureManager`) zostaje gdzie jest — bo ICultureManager nie potrzebuje RouteValues.
3. **Opcja C: `MapWhen("/{culture:regex(...)}/{**path}")` branch** — najczystsze, ale wymaga przepisania routingu Site/Area.

Po sprawdzeniu kodu: obecny `CultureMiddleware.ResolveCulture` nie używa `RouteValues` ani `HttpContext.GetEndpoint()`, więc **Opcja B jest najtańsza i wystarczająca**. Jeśli kiedyś trzeba będzie odczytać route values, migracja do A/C.

## 1.6. `SessionAuthenticationService` nie powiadamia o zmianie state'u

`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions.Website/Authentication/SessionAuthenticationService.cs:12-19`

```csharp
internal sealed class SessionAuthenticationService(IAuthenticatedProvider authenticated)
    : AuthenticationStateProvider
{
    public override Task<AuthenticationState> GetAuthenticationStateAsync()
    {
        var principal = IdentityClaimsBuilder.BuildPrincipal(authenticated.Current);
        return Task.FromResult(new AuthenticationState(principal));
    }
}
```

Brak subskrypcji na `IAuthenticatedRepository.Initialize` (i tak `IAuthenticatedRepository` nie ma takiego eventu). Po loginie/logoucie w długo żyjącym Blazor Server circuit, `<AuthorizeView>` i wszystkie auth-aware komponenty **nie odświeżą się** dopóki nie wymusimy reload. Brak `NotifyAuthenticationStateChanged()`.

W architekturze hydratacja idzie przez `SessionMiddleware` per request, ale **Blazor circuit nie hydratuje per request** (długo żyje) — dlatego notyfikacja jest konieczna.

**Naprawa** — dwie drogi:

1. **Opcja A (audyt pierwotnie): event w `IAuthenticatedRepository.OnChange`**, podpiąć z `SessionAuthenticationService` → `NotifyAuthenticationStateChanged()`. Plus: spójne z `ICultureManager.OnChange`, `IWorkspaceManager.OnChange`. Minus: każdy może zainicjować zmianę state'u (`Initialize` jest publiczne).
2. **Opcja B (czystsze)**: ujawnić login-flow API w `IAuthSource` (np. `Task<Identity> SignInAsync(...)`), który po sukcesie sam wywoła `NotifyAuthenticationStateChanged` przez wstrzyknięty `AuthenticationStateProvider`. Zachowuje SRP — repository pozostaje "data holder", scheme/auth-state-provider pozostaje "notifier". Tylko login flow ma prawo zmienić state.

**Decyzja**: Opcja B jest preferowana długoterminowo, ale Opcja A jest najmniejszą zmianą do natychmiastowego unlock. Można zacząć od A, potem zrefaktorować do B przy okazji audytu auth flow.

## 1.9. `PageViewBase.PersistState` / `TryTakeModelFromState` — fake AOT z błędną justyfikacją

`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions.Website/Components/PageViewBase.cs:144-164`

Justification:

> ".NET 11 doda przeciążenie PersistAsJson(JsonTypeInfo) i wtedy supresję można usunąć."

**To nieprawda** — `PersistentComponentState.PersistAsJson<TValue>(string key, TValue instance, JsonTypeInfo<TValue> jsonTypeInfo)` **istnieje już w .NET 9+** i jest dostępne w .NET 10. `ViewModelMetadata<T>` ma już property `JsonTypeInfo`:

## 1.11. `Permission.Implies` — semantyka wildcarda przy końcu prawej [🔵 — demoted, debatable]

> **Re-rated 🔴 → 🔵 (2026-05-15)**: Po review nie jest to clear-cut bug, tylko **design choice**. Większość systemów RBAC (AWS IAM: `s3:*` ⊇ `s3:ListAllBuckets`; K8s RBAC: `verbs: ["*"]` obejmuje wszystkie) traktuje `*` jako "wszystkie sub-actions ORAZ hosta". `@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/Auth/Permission.cs:84-105`

```csharp
public bool Implies(Permission other)
{
    ...
    for (int i = 0; i < Math.Max(left.Count, right.Count); i++)
    {
        var l = i < left.Count ? left[i] : null;
        var r = i < right.Count ? right[i] : null;

        if (l == Wildcard) continue;       // wildcard matches anything
        if (l is null) return false;
        if (r is null) return false;
        ...
```

| Semantyka | `"admin.*".Implies("admin")` | Przykład w branży |
|---|---|---|
| Wildcard = "≥1 token" (strict) | `false` | rare |

**Naprawa = decyzja designerska + dokumentacja**:

1. **Jeśli zostawiamy `true`** (rekomendowane — zgodne z industry standard): dodać XML doc na `Implies` z explicit example: `"admin.*".Implies("admin") == true` i podlinkować RFC/AWS IAM spec.

**Test wymagany**: niezależnie którą semantykę przyjmiemy, dodać test cases pokrywające macierz wildcard×length, żeby nie regredować.

## 2.1. `ValueObjectTypeConverter<T>` — heavy reflection na hot path

# 7. Kompatybilność po breaking-change (🟡)

## 7.1. `Identity` to `readonly struct` — boxowanie w niektórych ścieżkach

Wszystkie wywołania `Identity.HasValue`, `Identity.HasPermission` na `IAuthenticatedProvider.Current` są OK (struct copy by value). Ale `Identity` przekazywane przez `object?[] args` w log statementach (`Logger.LogInformation("user {User}", identity)`) → boxowanie.

## 7.2. `ImmutableArray<Permission>` w `Identity`

Każde `identity.Permissions` zwraca array. OK pattern. Ale `identity.HasPermission(...)` foreach przez tablicę — dla użytkownika z 50+ perm'ów × multiple checks per request → CPU overhead.

**Naprawa**: cached `FrozenSet<Permission>` w Identity (po hydration), wewnątrz konstruktora. Obecny `ImmutableArray` pozostawić w API.

## 7.3. Implicit conversions z `Guid` do `Identity`/`Organization`/`Project`

`Identity i = someGuid;` działa. `int x = Identity.Empty.Id;` nie skompiluje (nie ma `int` konwersji). Logiczne. Ale:

```csharp
public Identity? Author { get; init; }   // nullable struct
Author = Guid.NewGuid();   // ← działa: implicit Guid → Identity → Identity?
```

Wszystko OK. Ale następująca pułapka:

```csharp
if (entity.Author.Id == Guid.Empty) { ... }   // ← Author == default(Identity), Id getter → Guid.Empty
```

Identity.Empty ma `Id == Guid.Empty`. OK. Ale `Identity.Empty.HasValue == false` to bezpieczniejszy check.

## 7.4. `IWorkspaceProvider.IsPermission`/`IsRole` usunięte — konsumenci muszą migrować

Memory potwierdza: "IsPermission/IsRole **usunięte** — autoryzacja w Zonit.Extensions.Auth, single source of truth". OK breaking change, ale brak dokumentacji migracyjnej. Konsumenci mogliby dostać deprecated metodę z `[Obsolete("Use [RequirePermission] attribute instead")]` przed pełnym usunięciem.

## 7.5. `ICatalogProvider.Visible : ImmutableArray<Project>` — konsumenci pewnie mają `IList<Project>`

ImmutableArray ma trochę inny API niż List. `Visible.Count` działa, `Visible.IndexOf(...)` działa, ale `.Add(...)` nie. Konsumenci po LINQ nie powinni mieć problemów.

## 7.6. `Identity.Roles` / `Permissions` jako `ImmutableArray<T>`

Konsumenci robiący `foreach` OK. Konsumenci robiący `.Count > 0` na `default(Identity)` → `ImmutableArray<T>.IsDefault` ale getter zwraca `ImmutableArray<T>.Empty` (jak w `Identity` line 58-62), dobrze.

## 7.7. Stary namespace `Zonit.Extensions.Organizations.Abstractions.Entities`

(opisany w 2.14). Rage przy używaniu w istniejącym EF mappingu.

## 7.8. `UserModel.Policy` (Lista<string>) vs `Identity.Permissions` (Lista<Permission>) — dwa źródła

Konsumencki adapter `IAuthSource.GetByIdAsync` zwraca `UserModel?` z `Policy`, ale `IAuthSource.GetByTokenAsync` zwraca `Identity` z `Permissions`. Te dwie ścieżki mogą się rozjechać.

**Naprawa**: ujednolicić — albo całkowicie usunąć `UserModel` (skoro `Identity` jest the source of truth), albo `UserModel.Policy: List<Permission>`.

## 7.9. `IAuthenticatedRepository.Initialize` jest publicznym method — bezpieczeństwo

(opisane w 1.6) Każdy z DI access może tę metodę wywołać. Powinien być wewnętrzny lub przez `IAuthenticationContext` marker interface którego tylko middleware może uzyskać.

# 11. Uzupełnienia po review (post-audit findings)

Dodane na podstawie review właściciela repo — pominięte w pierwszej iteracji lub odkryte przy okazji korekt.

## 11.1. `MD5` w `Asset` — deprecated hash, fake security 🟡

`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/Files/Asset.cs:248-251, 308-310`

```csharp
Sha256 = data.Length > 0 ? Convert.ToBase64String(SHA256.HashData(data)) : string.Empty;
Md5    = data.Length > 0 ? Convert.ToBase64String(MD5.HashData(data))    : string.Empty;
```

XML doc na `Md5` (linia 122-124):

> "MD5 hash of the file data (Base64 encoded). Computed once at creation. Use for legacy systems, ETags."

Komentarz na klasie (linia 19): `"SHA256 and MD5 hashes for integrity and deduplication"`.

**Problemy**:

1. **MD5 jest cryptographically broken** od ~2005 (collisions trywialne). NIST formal deprecation 2010, RFC 6151 (2011) odradza. Każdy use case "integrity" jest fake security — atakujący może stworzyć dwa pliki z tym samym MD5.
2. **`MD5.HashData` na 100MB asset = ~200ms CPU dodatkowo** ponad SHA256 — bez wartości.
3. **Storage overhead**: 24 dodatkowych bajtów per Asset w binarnym formacie (`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/Files/Asset.Storage.cs:84-88`).
4. **`VerifyMd5` API w publicznym surface** (`Asset.cs:486-487`) — sugeruje że MD5 jest "supported alternative" do SHA256. Konsumenci użyją na produkcji.

**Naprawa** — dwie opcje:

1. **Usunąć MD5 całkowicie** (preferowane, ale breaking change w binary format): bump `StorageVersion`, removed Md5 z public surface, dodać `[Obsolete("MD5 is cryptographically broken. Use Sha256 for integrity.")]` na deprecated API by smooth migration.
2. **Zostawić ale jasno doc'ować**: zmienić XML doc na `Md5` z "Use for legacy systems, ETags" na `[Obsolete]` + warning że ma TYLKO use case "legacy interop, NOT integrity". Usunąć z sentence o "integrity" w klasowym summary.

## 11.2. `ToLowerInvariant` pod NativeAOT — culture-aware ops 🔵

Wzory wzorcowe (znalezione w 12 miejscach):

`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/Auth/Permission.cs:67`
`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/Auth/Role.cs:47, 89`
`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/Auth/Credential.cs:147`
`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/Text/UrlSlug.cs:60, 87`
`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/Files/Asset.MimeType.cs:194, 224`

```csharp
var normalized = value.Trim().ToLowerInvariant();
```

**Kontekst**: `string.ToLowerInvariant()` jest **AOT-safe** — używa `CultureInfo.InvariantCulture` i runtime CharUnicodeInfo data, które są wbudowane w runtime. **Action: nic do naprawy**. Zostawiam jako finding bo user explicit pytał czy bezpieczne.

**Potencjalna optymalizacja** (🔵 nice-to-have): dla ASCII-only inputs (jak `Role`, `Permission` ograniczone do `[a-z0-9_-]+`) można użyć szybszego ASCII-only path:

```csharp
// dla VOs gdzie regex już ogranicza do ASCII:
var normalized = string.Create(value.Length, value, (span, input) =>
{
    for (int i = 0; i < input.Length; i++)
        span[i] = (char)(input[i] | 0x20);  // ASCII tolower trick
});
```

Eliminuje culture lookup table cost. **Niski impact** w hot-path (per-request perm check) bo i tak normalizacja idzie raz w ctor. Można zostawić tak jak jest.

## 11.3. `UrlPath` VO — edge cases nie pokryte 🟡

`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/Text/UrlPath.cs`

VO świeżo wprowadzone, brak pokrycia w pierwszej iteracji audytu.

**Problemy**:

1. **`TryNormalize` nie normalizuje** — tylko waliduje. `"/foo/bar/"` przechodzi as-is (trailing slash). `"/foo//bar"` przechodzi (double slash). `"/foo/../bar"` przechodzi (path traversal). `"/foo?q=1&q=2"` przechodzi (duplicate query keys).
2. **Brak `MaxLength` w `TryCreate`** (linia 87-103) — sprawdza tylko w `TryNormalize` (linia 112). Ale `string.IsNullOrWhiteSpace` skraca do Empty bez sprawdzenia długości. ✅ Faktycznie OK — `TryNormalize` jest wywoływane przez `TryCreate`.
3. **Asymetria ctor vs implicit operator**:
   ```csharp
   var p1 = new UrlPath("//cdn/foo");   // throws ArgumentException
   UrlPath p2 = "//cdn/foo";            // → UrlPath.Empty (silent)
   ```
   Identyczny pattern jak w `Title`, `Url`, `Culture` — niespójny i tracimy informację o błędzie przez cast.
4. **`Value` getter zwraca `string.Empty` dla default**:
   ```csharp
   public string Value => _value ?? string.Empty;   // ← OK
   public bool IsRooted => HasValue && _value![0] == '/';   // ← null-forgiving OK bo HasValue gwarantuje
   ```
   ✅ Bezpieczne.
5. **`ToString(string? pathBase)` allokuje 2 stringi w niektórych ścieżkach**:
   ```csharp
   var trimmedPath = IsRooted ? _value!.AsSpan(1).ToString() : _value ?? string.Empty;
   return trimmedPath.Length == 0 ? trimmedBase + "/" : trimmedBase + "/" + trimmedPath;
   ```
   `AsSpan(1).ToString()` alokuje pierwszy string, potem concat alokuje drugi. Można `string.Concat` z `ReadOnlySpan<char>`:
   ```csharp
   return _value.AsSpan(IsRooted ? 1 : 0) is var path && path.IsEmpty
       ? string.Concat(trimmedBase.AsSpan(), "/")
       : string.Concat(trimmedBase.AsSpan(), "/", path);
   ```
6. **`UrlPathJsonConverter` jest poza factory `CanConvert`** (`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions/ValueObjects/ValueObjectJsonConverterFactory.cs`) — już wymieniony w 2.12, ale wart wzmianki w kontekście `UrlPath` bo to ostatnio dodany VO.

**Naprawa**: ujednolicić politykę normalizacji (rozważyć: czy `UrlPath` powinien deduplicate slashes? Reject path traversal? Sortować/dedup query keys?). Decyzja designerska — zależy od use case.

## 11.4. `WebsiteAreaRegistry` re-registration semantics 🔵

`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions.Website/Areas/WebsiteAreaRegistry.cs:24-28`

```csharp
public T Register<T>(T instance) where T : notnull
{
    ArgumentNullException.ThrowIfNull(instance);
    return (T)_instances.GetOrAdd(typeof(T), instance);
}
```

**Pozytywne**: `ConcurrentDictionary.GetOrAdd` jest thread-safe i idempotentne. Jeśli `AddWebsite(o => o.AddArea<DashboardArea>())` zostanie wywołane 2× (np. raz w `Program.cs`, raz w testach przez `WebApplicationFactory.WithWebHostBuilder`), pierwsza rejestracja wygrywa.

**Subtle problem**: jeśli druga rejestracja przekaże **inną** instancję (np. inny config przez ctor `new DashboardArea(otherConfig)`), **zostanie cicho zignorowana**. Brak warning'u, brak exception. Konsument może spodziewać się że druga rejestracja "wygrywa" lub że dostanie diagnostic.

**Naprawa** (low priority — pattern jest dokumentowany jako "first-wins"):

1. Dodać XML doc na `Register<T>` mówiący jasno "first instance wins; subsequent registrations are silently ignored".
2. **Lub** w `Debug` configuration `Debug.Assert(_instances[typeof(T)] == instance, "...")` — pomaga w developmencie wyłapać accidental double-registration.

**Inny related point** — `WebsiteOptions.AddArea<TArea>()` (`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions.Website/Options/WebsiteOptions.cs:60-72`) tworzy `new TArea()`, więc każde wywołanie `AddArea<T>()` powoduje nową instancję `T` ALE tylko pierwsza jest faktycznie wstrzykiwana (przez registry GetOrAdd). Drugi `new TArea()` to garbage. Trochę wasteful jeśli ctor robi dużo pracy. **Naprawa**: `_registry.Contains<T>() ? _registry.Resolve<T>() : _registry.Register(new T())`.

## 11.5. `SiteOptions.AppHooks` / `UseHooks` / `EndpointHooks` idempotency 🔵

`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions.Website/Options/SiteOptions.cs:23-25, 142-147`
`@/d:/GitVsCode/Zonit.Sdk/Source/Extensions/Zonit.Extensions/Source/Zonit.Extensions.Website/DependencyInjection/WebsiteServiceCollectionExtensions.cs:306-307`

```csharp
foreach (var area in site.Areas) area.App(branch);
foreach (var hook in site.AppHooks) hook(branch);
```

**Pozytywne**: Każdy site ma własną instancję `SiteOptions` (per `o.AddSite(...)` w `UseWebsite`), więc hooks są per-site izolowane.

**Subtle problem**: te same `IWebsiteArea` instances są shared między site'ami (przez `WebsiteAreaRegistry`). Jeśli `Area.App(IApplicationBuilder)` ma side effects (dodaje middleware, rejestruje routes), będzie wykonane **N razy** dla N sites. Czy to intended?

```
site1.UseRouting().UseAuthentication()...
  for each area: area.App(site1_branch)   // Dashboard.App(branch1) — adds middleware X
site2.UseRouting().UseAuthentication()...
  for each area: area.App(site2_branch)   // Dashboard.App(branch2) — adds middleware X again
```

Każdy site ma własną pipeline, więc duplikacja middleware jest fine — middleware nie jest singleton, jest per-branch. **Ale** jeśli `Area.App` robi coś globalnego (np. zapisuje state w singleton service), będzie wywołane N razy.

**Naprawa** (low priority): dokumentacja w XML doc `IWebsiteArea.App` mówiąca jasno "called once per mounted Site — side effects against singleton services WILL run N times, plan accordingly". Plus jeśli area chce "register once globally", powinno to być w `IWebsiteServices.ConfigureServices` (które jest gwarantowanie raz).
